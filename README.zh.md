# idea-wiki

> 一个把 **个人想法、研究调研、认知对齐、知识沉淀** 放进同一闭环里的 idea-driven research workflow。💡

`idea-wiki` 不是普通的笔记仓库，也不是单纯的灵感收集箱。
它要解决的是一个更具体的问题：

> **当人的脑中出现一个模糊想法时，怎样把它一步步变成可研究的问题，再与已有工作对齐，并最终沉淀成稳定理解？**

这个仓库最适合和 **CLI code agent** 一起使用。🤖

---

## 🚀 怎么用

### 推荐使用方式

推荐的基本 setup 是：

- 把这个仓库放在本地；
- 配合一个可以直接读写仓库文件的 **CLI code agent** 使用（例如 Codex CLI、Claude Code，或其它同类本地 agent）；
- 让 agent 帮你梳理想法、引入资料、深挖来源，并把结果写回仓库。

为什么推荐 CLI agent？
因为这套 workflow 是文件原生的。agent 最好能直接：

- 读取你当前的 `problems`、`concepts`、`sources`；
- 更新页面、链接和结构；
- 把 raw source、研究笔记、问题空间和 ideas 放在同一个连续闭环里。

### 推荐工作流

1. 从一个问题、直觉或半成型想法开始；
2. 用 `brainstorming` 把它整理成更清楚的问题定义；
3. 用 `research-intake` 引入相关来源；
4. 用 `source-study` 深挖关键材料；
5. 只把真正稳定的理解提升到 `concepts` 或 `problems`；
6. 再让新的问题和 `idea_seeds` 驱动下一轮研究。

换句话说，不要把这个仓库当成静态储物柜。
要把它当成一个持续运转的研究闭环：

**idea -> research -> alignment -> synthesis -> next idea**

## 💡 为什么要把想法和学习放在一起

单纯做知识管理，常见的问题是：你会积累很多材料、很多笔记、很多总结，但它们不一定真的在推进你的问题。
你知道了很多东西，却不一定更接近自己真正想研究的方向。

单纯记录想法，也有另一种问题：你会有很多直觉、很多设想、很多“这里可能有东西”的感觉，但如果没有持续学习和外部研究来校正，这些想法很容易停留在主观判断里。

于是就出现了两种断裂：

- **只有知识，没有主线**：学了很多，但没有形成自己的研究推进。📚
- **只有想法，没有校正**：想了很多，但不知道它与现有工作的距离。🤔

`idea-wiki` 要做的，就是把这两件事接起来：

- 让 **idea 驱动学习**，而不是无方向地囤积知识；
- 让 **学习反过来修正 idea**，而不是让想法停留在未经检验的直觉里。

它关注的不是“记住了什么”，而是：

> **你的原始想法，怎样通过研究逐步变得更清楚、更可靠、更接近现实，也更接近已有工作。**

## 🧭 这套框架在做什么

这个仓库服务的是一个持续循环，而不是一次性的记录动作：

```text
idea -> structure -> research -> alignment -> synthesis -> new idea
```

每一段都有不同职责：

1. **idea**  
   从一个问题、直觉、猜想或方向感开始，哪怕它还很模糊。

2. **structure**  
   把这个模糊想法整理成可讨论、可研究的问题空间：它到底在问什么？缺什么？属于什么问题类型？

3. **research**  
   再引入论文、案例、已有工作和外部材料，让这个想法不再只是私人直觉。

4. **alignment**  
   把“我脑子里的表达”与“现有研究真正如何表述这个问题”对齐。很多偏差、空白与错位会在这里暴露出来。

5. **synthesis**  
   当某种理解跨过单一来源、能稳定成立时，再把它沉淀成概念、问题页和知识关系。

6. **new idea**  
   沉淀不会结束研究，反而会长出新的问题、新的方向，以及更好的下一轮想法。

所以 `idea-wiki` 的重点不是“存资料”，而是：

> **让一个人的想法，有机会进入研究；让研究结果，又能反过来改变这个人的想法。**

## 🤖 AI 在这里扮演什么角色

在这套框架里，AI 不是用来替你下判断的，也不只是一个普通的摘要器。

更准确地说，它扮演的是一个 **研究对齐器（research aligner）**：

- 帮你把模糊想法整理成可研究对象；
- 帮你找出下一步缺什么知识；
- 帮你把个人直觉映射到更稳定的研究语言里；
- 帮你发现“你以为自己在问的问题”和“学界真正讨论的问题”之间的差距。🧠

很多研究推进的障碍，并不是“论文看得不够多”，而是：

- 问题没有被清楚表达；
- 调研没有围绕真正的问题展开；
- 个人语言和文献语言彼此错位；
- 读了很多材料，却没有回流到自己的原始问题。

AI 最有价值的地方，就是帮助你在这些断裂之间搭桥，而不是替你思考。

## 🧱 这个仓库如何承载这个闭环

这个仓库的结构不是为了分类而分类，而是为了让上面的 workflow 有稳定落点。

### `raw/`
保存原始材料本体。
原则是：**原始输入尽量不被污染**。
你后续的摘要、解释和概念化，不应该反写回原始材料层。

### `wiki/sources/`
这里存放来源笔记（source notes）。
它们的作用不是复制原文，而是把来源转成一个可复用的研究节点：摘要、关键论点、局部想法，以及和其它页面的可解释关系。

### `wiki/concepts/`
这里沉淀的是已经跨过单一来源、具有更稳定意义的概念。
如果一个理解只在某个单独来源里成立，它不应该太早被提升到这里。

### `wiki/problems/`
这是最关键的层之一。
这里不是任务清单，而是 **问题空间 / idea space**：

- 当前真正重要的长期问题是什么；
- 还缺哪些知识；
- 正在长出哪些 `idea_seeds`；
- 下一步应该读什么，才能让问题更清楚。

这里的核心作用，就是持续把“想法”和“学习”接在一起。

### `skills/`
这里放的不是零散工具，而是配合这套闭环使用的 workflow skills。
它们帮助你进入不同环节，而不是打断流程。

## 🛠️ 内置技能

当前仓库内置了三个技能：

### `brainstorming`
在真正动手前，先把一个模糊方向梳理成更清楚的设计或问题定义。
它对应的是从 **idea** 进入 **structure** 的环节。

### `research-intake`
当你已经有一个问题或方向，但还缺外部材料时，用它来筛选来源并把高价值材料引入仓库。🔎
它对应的是从 **structure** 进入 **research** 的环节。

### `source-study`
当一个本地来源值得深入学习时，用它来进行深度研究、更新来源笔记，并保守修复周边图谱。
它对应的是从 **research** 进入 **alignment / synthesis** 的环节。

这三个技能拼在一起，构成的不是“自动化流水线”，而是一套更可控的研究辅助结构。

---

## 🚫 这不是什么

为了避免误解，`idea-wiki` **不是**：

- 一个传统笔记仓库；
- 一个任务管理器；
- 一个“把所有资料都收进来”的大档案馆；
- 一个让 AI 替你完成最终判断的自动研究系统。

它更像是一种研究工作方式：

> **把人的问题意识、AI 的整理能力、外部知识的校正作用，以及长期沉淀的结构，放进同一个系统里。**

---

## 📚 仓库速览

- `raw/` — 原始材料与注册表
- `wiki/sources/` — 来源笔记
- `wiki/concepts/` — 稳定概念
- `wiki/problems/` — 带有 `idea_seeds` 的长期问题空间
- `skills/` — brainstorming / intake / study 的 workflow skills

## 🙌 灵感来源

这个项目 **受 Andrej Karpathy 的 `llm_wiki` 启发**。

不同的是，`idea-wiki` 想把这条路线再往前推进一步：
它不只组织知识，还显式组织 **ideas**、**problem spaces**，以及“个人直觉如何与已有研究逐步对齐”的过程。

可以把它理解成：在 `llm_wiki` 思路之上，一个加入了 idea-driven research 维度的升级版。
